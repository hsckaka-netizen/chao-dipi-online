# 炒地皮在线房间

第一阶段目标：先跑通真正多人在线房间的基础能力。

已实现：

- 创建房间并生成房间号。
- 玩家账号即牌桌身份：账号包含登录名、昵称、头像和牌局统计，创建或加入房间时直接使用当前登录账号。
- 支持管理员创建玩家账号并分别设置昵称、头像、头像框、牌面边框和出牌特效。
- 玩家可以修改自己的密码并上传头像，服务器限制每 7 天最多更换一次头像。
- 管理员可以管理昵称、头像、账号启停、密码重置、头像框、牌面边框和出牌特效。
- 创建房间、加入房间和观战均需要登录；未登录可查看房间列表和数据榜。
- 首页提供房间列表与数据模块，创建房间为快捷操作，加入房间只需输入房间号。
- 数据模块提供完整排行榜、玩家个人数据、积分走势和赛季筛选；管理员可以创建赛季并设置起止时间和当前赛季。
- 当前支持 5-9 人开始。
- 房主可在开局前选择叫庄方式和狗腿数量，狗腿数量不超过总人数 - 3。
- 房主开始后，系统按人数生成对应副数的完整扑克。
- 每人发 53 张牌。
- 底牌数量等于人数。
- 每个玩家页面只收到自己的手牌。
- 房间状态通过浏览器实时事件同步。
- 房主可以重复点击添加机器人，每次添加 1 个机器人，机器人可被踢出。
- 发牌后进入叫主/抢主，支持选择同花色的 2 叫主或抢主。
- 如果无人叫主，房主可以触发随机指定主。
- 每次亮主、抢主和炒底后，页面会展示当前主牌花色。
- 主拿底后需要选择同数量牌贴底。
- 主拿底或玩家炒底拿底后，新进入手牌的底牌会保持选中，方便识别。
- 贴底后会停在炒底阶段，玩家可以用更大的 2 炒底，也可以选择不炒。
- 叫主/抢主和炒底阶段会显示每个玩家的等待、过、当前操作状态。
- 机器人会自动处理叫庄、炒底、贴底、选狗腿和出牌流程，遇到真人操作时停下。
- 炒底结束后按最后一次亮出的 2 确定主牌。
- 主可以选择 1 张非比牌作为狗腿牌，然后进入打牌。
- 玩家列表会标注主、已暴露的狗腿和闲家；狗腿牌被打出后会自动识别狗腿。
- 玩家信息会显示得分、被拖红五、被拖方五和甩牌失败次数。
- 发牌后显示所有玩家、当前轮出牌、历史出牌。
- 玩家可以从自己的手牌中选择牌并出牌。
- 准备阶段参谋 3 暂不进入比牌，仍按花色放置。
- 打牌阶段，主花色普通牌、主花色 3、同颜色另一花色 3 会和比牌放在同一个“主牌/比牌”分组，并按主牌牌力排序。
- 手牌支持折行显示，主牌/比牌很多时不会只依赖横向滚动。
- 手牌支持点击选择、按住滑动选择多张、点击空白处取消选择。
- 首家出牌已校验基础牌型：单张、同点数多张、同花色连续多张拖拉机。
- 后手跟牌不限制牌型，只校验必须同张数，且优先跟首家花色。
- 本轮结束后会按同牌型、同首家花色的牌比较大小，胜者获得本轮 5/10/K 分并成为下轮首家。
- 比牌/主牌拖拉机支持按主牌牌力相邻组合，例如红五+方五、大小王、主牌 A+副 2。
- 真人玩家出牌后，后续轮到的机器人会自动出牌，直到轮到下一个真人。
- 如果一轮最大者是机器人，它会自动开启下一轮并继续出牌，直到轮到真人。
- 支持首家甩牌：成功时多手牌型一起打出；失败时自动改出被压过的最小牌型，并记录甩牌失败用于结算。
- 最后一轮打完后会自动进入结算，展示闲家目标分、保底、拖五、闲家/庄家每人积分和玩家明细。
- 打牌阶段会固定展示狗腿牌，并在玩家打出狗腿牌、身份变成狗腿时提示所有玩家。
- 准备阶段退出会真正移除玩家；牌局中可暂离，同一登录账号可从房间列表返回原席位、手牌和操作权。
- 红桃 5、方块 5 一旦在出牌过程中确定不再是当前最大，会立即显示被拖动效，无需等待整轮结束。

暂未实现：

- 服务器重启后的进行中牌局恢复。

## 玩家账号与头像

账号密码由 Supabase Auth 保存。业务上账号与玩家是同一实体，项目数据库以 `account_id` 关联牌局历史；`profile_id` 仅作为现阶段兼容既有牌局代码的内部资料编号，不再作为用户可选择的独立身份。Supabase Secret Key 只允许配置在服务器环境变量中，不能写入前端文件。

环境变量：

```text
SUPABASE_URL=https://你的项目.supabase.co
SUPABASE_SECRET_KEY=Supabase Secret Key
AUTH_SESSION_SECRET=一段独立的随机长字符串
ADMIN_BOOTSTRAP_USERNAME=首次管理员用户名
ADMIN_BOOTSTRAP_PASSWORD=首次管理员初始密码
```

首次上线时填写 `ADMIN_BOOTSTRAP_USERNAME` 和 `ADMIN_BOOTSTRAP_PASSWORD`。管理员创建成功并确认可以登录后，应从服务器环境变量中删除这两项；数据库里的管理员账号不会因此删除。

头像在浏览器中裁切并压缩为 256×256、最多 300KB，再上传到 Supabase Storage 的 `player-avatars` 公共桶。每次头像使用新的版本地址，可以长期缓存而不会显示旧图。普通玩家的 7 天限制由服务器和数据库时间共同判断，不能通过修改页面绕过；管理员代为更换头像不受该限制。

## 牌局历史（分阶段启用）

项目支持把现有结算结果旁路写入 PostgreSQL。该功能不参与牌局判断，数据库不可用时不会阻断开局、出牌或结算。

环境变量：

```text
DATABASE_URL=Supabase Session pooler 连接地址
GAME_HISTORY_ENABLED=false
```

只配置 `DATABASE_URL` 时，服务启动会创建或升级数据表，并持久化预置玩家的名称、头像框、牌面边框和出牌特效，但不会保存正式牌局。完成连接验证后，将 `GAME_HISTORY_ENABLED` 改为 `true` 才会开启牌局影子记录。

牌局过程中不会读写历史数据库。只有全部参与者都是真人且均已登录玩家账号的牌局，才会在最后一轮结算完成后把结果放入一次写入任务；有任意机器人参与的牌局不会记录。同一 `game_id` 重复触发时数据库会自动忽略，避免重复牌局和重复计分。出牌历史只保存牌的编号和必要的甩牌信息，玩家和标签使用批量写入，减少数据库连接、请求次数和存储体积。

数据结构：

- `cdp_games`：牌局、规则、庄闲胜负、底牌和完整出牌历史。
- `cdp_game_players`：每位玩家的阵营、角色、牌分、积分和拖五数据。
- `cdp_game_tags`：MVP、躺、坑、辅、僵等牌局标签。
- `cdp_player_statistics`：优先按玩家账号汇总的兼容统计视图，机器人不计入。
- `cdp_player_profiles`：账号昵称、头像版本、头像框、牌面边框和出牌特效的内部资料表。
- `cdp_accounts`：玩家登录名、Supabase Auth 用户编号和账号权限。
- `cdp_seasons`：赛季名称、开始/结束时间和当前赛季标记，由管理员维护。

只读检查接口：

```text
GET /api/history/status
GET /api/history/statistics
GET /api/history/seasons
GET /api/history/players/:accountId
GET /api/history/games?limit=30
```

## 本地启动

```bash
cd tools/chao-dipi-online
npm start
```

打开：

```text
http://localhost:3000
```

同一台电脑上可以用多个浏览器窗口模拟多个玩家。
